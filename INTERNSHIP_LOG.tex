\section{Challenges Faced \& Solutions}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{Challenge 1: SMTP Port Blocking on Render}

\begin{itemize}
  \item \textbf{Problem:} Email sending failed because Render blocks SMTP port 587.
  \item \textbf{Solution:} Implemented the Resend REST API as an alternative to SMTP.
  \item \textbf{Outcome:} Email delivery now reliable on production with detailed delivery
        logs and connection timeout handling.
\end{itemize}

\subsection{Challenge 2: Patient Appointments Crash}

\begin{itemize}
  \item \textbf{Problem:} Demo patient creation caused the appointments listing to crash.
  \item \textbf{Solution:} Fixed patient profile seeding logic and validation rules.
  \item \textbf{Outcome:} Demo flow is now stable and repeatable across resets.
\end{itemize}

\subsection{Challenge 3: Browser Caching Issues}

\begin{itemize}
  \item \textbf{Problem:} Doctor verification changes did not reflect due to stale cache.
  \item \textbf{Solution:} Implemented proper cache-busting headers on relevant GET endpoints.
  \item \textbf{Outcome:} All GET requests now always return fresh data.
\end{itemize}

\subsection{Challenge 4: Financial Data Consistency}

\begin{itemize}
  \item \textbf{Problem:} Refund processing would incorrectly reset entire payment history
        and earnings figures.
  \item \textbf{Solution:} Implemented proper transactional logic and state isolation between
        the refund and earnings pipelines.
  \item \textbf{Outcome:} Financial data is now consistent, auditable, and safe to process
        across concurrent operations.
\end{itemize}

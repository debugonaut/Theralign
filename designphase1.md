# PROMPT D1.1 — Tailwind Configuration & Design Token System

## Objective
Replace the existing Tailwind configuration with a complete, Swiss-adapted design token system. Every color, typography scale, spacing value, border width, and animation timing used across all 5 design phases is defined here and nowhere else. No component in any subsequent phase introduces values that are not in this token file.

## Why This Exists
The current `tailwind.config.js` from Phase 1 of engineering has functional tokens that served the MVP build. They are not wrong — they are incomplete for a production design system. The redesign requires a single authoritative token source because five phases of frontend work will be generated by AI tools. Without locked tokens, those tools will introduce arbitrary values — a `rounded-lg` here, a `shadow-md` there — that silently break the system's visual authority. The config file is the contract that prevents drift.

## Architecture Reasoning
Swiss Minimalist adapted for healthcare SaaS means: Inter at all weights, near-monochrome palette, exactly two signal colors (red and teal), zero border radius except avatar circles, no drop shadows, visible structural borders, and mechanical micro-interactions. Every token decision below is justified against this philosophy and the guardrails document.

## Expected File
`client/tailwind.config.js` — complete replacement

## Complete Token Specification

### Colors
```javascript
colors: {
  // Base surfaces
  'swiss-white':   '#FFFFFF',   // Primary background — pure canvas
  'swiss-black':   '#0F0F0F',   // Primary text — not pure #000, one degree warmer
  'swiss-gray': {
    50:  '#FAFAFA',             // Lightest surface — barely off-white
    100: '#F2F2F2',             // Muted surface — card backgrounds, sidebar panels
    200: '#E5E5E5',             // Light border — internal table row separators only
    400: '#A3A3A3',             // Secondary text — labels, metadata, timestamps
    600: '#525252',             // Body text on gray surfaces
    900: '#171717',             // Near-black for high-contrast text on white
  },

  // Signal colors — used in EXACTLY the permitted contexts
  'swiss-red':  '#FF3000',      // CTAs, active nav, section prefixes, destructive actions
  'swiss-teal': '#0D7377',      // Verified badges, confirmed status, paid status ONLY

  // Status colors — expressed through the monochrome system
  'swiss-amber': '#B45309',     // Pending state — dark amber, never bright yellow
  // Cancelled uses swiss-gray-400 — gray IS the cancelled signal
}
```

**Why `#0F0F0F` not `#000000`?**
Pure black on white creates maximum contrast but reads as cold and clinical on screen. `#0F0F0F` is imperceptibly warmer but prevents the sterile feeling that would undermine healthcare trust. This is the single degree of warmth referenced in the design philosophy.

**Why dark amber `#B45309` not bright `#F59E0B`?**
Bright amber in a healthcare context triggers alarm. Dark amber communicates "attention needed" without urgency. Patients seeing a pending status should feel informed, not alarmed.

Large type at default tracking feels loose and amateurish. Negative tracking at large sizes creates mass, authority, and the compressed visual weight that makes Swiss headlines feel like buildings rather than words.

# PROMPT D1.3 — Button Primitive Component

## Objective
Build the `Button` component with all four variants (`primary`, `secondary`, `accent`, `ghost`), all states (`default`, `hover`, `active`, `disabled`, `loading`), and consistent sizing. This is the most reused component in the product. Every interactive CTA, form submit, filter control, and action link that requires a button shape uses this component. It must be perfect.

## Why This Exists
The button is the primary conversion mechanism in a SaaS product. Patients click the book button. Doctors click the save profile button. Admins click the approve button. Every variant of button is a different hierarchy signal — primary for the most important action on a screen, secondary for the subordinate action, accent for the critical CTA (like Confirm & Pay), ghost for tertiary options. Getting this wrong in a single component means every page that uses it is wrong.

**Why label above, never inside?**
Placeholder-as-label is a UX failure documented across accessibility guidelines and usability research. When the patient starts typing, the label disappears — they can no longer see what they're filling in. **Why 4px border on focus, not a color change?**
Color changes are inconsistent across contexts — a color-focused state might clash with error states or validation states. A border weight increase is a pure structural signal: the element is active. It works in all contexts without color conflict.

```
completed:  black border (#0F0F0F), black text, white fill
            Default label: 'COMPLETED'
            Contexts: appointment completed, session done

active:     black border (#0F0F0F), black text, white fill
            Default label: 'ACTIVE'
            Contexts: user account active

neutral:    gray-200 border (#E5E5E5), gray-400 text, white fill
            Default label: custom via label prop

```

```

```

Filled colored badges (green background, red background) add visual noise to tables and lists. Bordered badges with white fill allow the content to remain primary — the status is signaled through border and text color without dominating.

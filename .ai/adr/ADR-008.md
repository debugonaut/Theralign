# ADR-008 — AppointmentMedia as Separate Collection with uploadedBy Field

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Appointments can have associated media (pre-booking patient documents, imaging, lab reports). Initially, media was embedded as an array in Appointment:

```javascript
Appointment.media = [
  { url: '...', uploadedAt: Date, uploadedBy: 'patient' }
]
```

However, media metadata was ambiguous:
- Who uploaded this media? (Patient? Doctor? Admin?)
- When was it uploaded relative to appointment time? (Before, during, after?)
- Can media be added/removed after appointment is completed?
- Should patient and doctor each see all media?

---

## Decision

AppointmentMedia is a separate collection with explicit `uploadedBy` field. Each media item is a separate document.

**AppointmentMedia model:**
```javascript
{
  appointment: ObjectId(Appointment),
  patient: ObjectId(User),
  doctor: ObjectId(DoctorProfile),
  
  media: [
    { url: String, publicId: String, type: String }
  ],
  
  uploadedBy: {
    type: String,
    enum: ['patient', 'doctor', 'admin'],
    required: true
  },
  
  uploadedAt: Date,
  
  description: String,
  isVisible: Boolean
}
```

---

## Consequences

1. **Explicit ownership:** `uploadedBy` field is always present. Code can check who uploaded and act accordingly.

2. **Audit trail:** Each media upload is logged with timestamp and uploader identity. Complete audit trail for medical compliance.

3. **Privacy control:** Patient-uploaded media is visible to doctor. Doctor-uploaded media (e.g., session notes) is visible to patient. Admin can see all.

4. **Mutable media:** Media can be added/updated/deleted after appointment completes. Separate collection allows this without modifying Appointment record.

5. **Separate queries:** Retrieving appointment does not fetch media. Media is fetched separately if needed.

6. **Media deletion:** When deleting media, can track what was deleted, who deleted it, and when (via audit trail).

---

## Alternatives Considered

**Approach 1: Embed media array in Appointment**
```javascript
{
  media: [{ url, uploadedBy, uploadedAt }]
}
```

*Rejected:* Ambiguous ownership. No audit trail of who deleted what.

**Approach 2: Embed media in AppointmentRecord (combined with SessionRecord)**
Store patient media, doctor media, and session notes in one place.

*Rejected:* SessionRecord is doctor-only; media is shared. Different authorization models break combined collection.

**Approach 3: Use single File collection for all uploads**
Generic File collection with `relatedTo: 'appointment'`.

*Rejected:* Too generic. AppointmentMedia has specific semantics.

## Implementation Notes

- AppointmentMedia model: `server/src/models/AppointmentMedia.model.js`
- AppointmentMedia controller: `server/src/controllers/appointmentMedia.controller.js`
- AppointmentMedia routes: `server/src/routes/appointmentMedia.routes.js`
- Upload service: Validates file type before creating AppointmentMedia
- Authorization: Patient can upload media for own appointments. Doctor can view but not upload.
- Indexes: `{ appointment: 1 }`, `{ patient: 1 }`, `{ uploadedBy: 1 }`
- API endpoints:
  - POST /api/appointment-media/:appointmentId — Upload media
  - GET /api/appointment-media/:appointmentId — List media for appointment
  - DELETE /api/appointment-media/:id — Delete media

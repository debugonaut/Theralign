# ADR-001 — Atomic Slot Locking via findOneAndUpdate

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Appointment booking in Theralign is time-sensitive. Multiple patients may attempt to book the same availability slot simultaneously. Without atomic locking, a race condition occurs:

1. Patient A queries slot (isBooked: false)
2. Patient B queries slot (isBooked: false)
3. Patient A marks slot booked and creates appointment
4. Patient B marks slot booked and creates appointment
5. Two patients are now booked for the same slot

Double-booking violates the appointment model and breaks refund logic.

---

## Decision

Slot locking uses MongoDB's `findOneAndUpdate` with a conditional query. The query includes both the slot ID and the condition `isBooked: false`. This is an atomic operation at the database level.

**Correct pattern:**

```javascript
const slot = await AvailabilitySlot.findOneAndUpdate(
  { _id: slotId, isBooked: false },
  { isBooked: true },
  { new: true }
);
if (!slot) {
  throw new AppError('Slot already booked', 409);
}
```

The operation either:
- Finds the slot, marks it booked, and returns the updated document (success)
- Finds no document because another transaction already marked it booked (race lost)

Both outcomes are deterministic. No slot can be double-booked.

---

## Consequences

1. **Slot unlocking:** If appointment creation fails after slot lock, the slot must be explicitly unlocked with another `findOneAndUpdate`:
   ```javascript
   await AvailabilitySlot.findOneAndUpdate(
     { _id: slotId },
     { isBooked: false },
     { new: true }
   );
   ```

2. **No transactions:** Transactions are not required. Each operation (slot lock, appointment create, payment) is independent and can fail without leaving the database in an inconsistent state. The booking flow is idempotent.

3. **Retry logic:** If slot lock fails, the patient should be offered alternative slots. Re-attempting the same slot will never succeed if another booking is in progress.

4. **Error handling:** Slot lock failure is not a 500 error. It is a 409 Conflict — the patient's chosen slot is unavailable. This is handled gracefully in the UI.

5. **Performance:** `findOneAndUpdate` is fast. No transactions, no two-phase commit. The operation completes in a single round-trip to MongoDB.

---

## Alternatives Considered

**Approach 1: Transactions**
Wrap slot lock + appointment creation in a multi-document transaction. If either operation fails, roll back both.

*Rejected:* Transactions add latency and complexity. The appointment creation can fail after the slot is locked (e.g., validation error, duplicate patient booking for same slot). Requiring a transaction would necessitate unlocking in a rollback, which is equivalent to the manual unlock pattern we already use.

**Approach 2: Separate find then save**
```javascript
const slot = await AvailabilitySlot.findById(slotId);
if (slot.isBooked) throw new Error('Slot taken');
slot.isBooked = true;
await slot.save();
```

*Rejected:* This is vulnerable to race conditions. Two processes can pass the `if (slot.isBooked)` check simultaneously, then both save the document. MongoDB's last-write-wins semantics mean the final saved document has `isBooked: true`, but the middle states are inconsistent.

**Approach 3: Optimistic locking with version field**
Add a `version` field to AvailabilitySlot. Increment on each update. Fail if version in update doesn't match current version.

*Rejected:* Over-engineered for this use case. `findOneAndUpdate` with conditional query is simpler and just as effective.

## Implementation Notes

- All slot locking code lives in `appointment.controller.js` in the `bookAppointment` handler
- Slot unlock code lives in error handlers (catch blocks) if booking creation fails
- Tests verify both success (slot locked) and failure (slot already booked) scenarios
- Monitoring alerts if slot lock failure rate exceeds 5% (indicates contention)

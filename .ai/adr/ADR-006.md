# ADR-006 — Fire-and-Forget Email Sending (Never Await in Critical Flows)

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Transactional emails (booking confirmations, cancellations, payment receipts) are sent on appointment and payment state changes. Email delivery can be slow:
- SMTP round-trip: 0.5–2 seconds typical
- Email service API: 0.2–1 second typical
- Network failures: can timeout and retry

If the booking flow awaits email sending:
```javascript
// WRONG — blocks patient interaction
await emailService.sendBookingConfirmation(appointment);
await Appointment.create(appointment);
```

Then a slow email server delays the patient's confirmation page by seconds. A network timeout causes the entire booking to fail.

---

## Decision

Email is sent asynchronously. The booking flow does not await email sending. The `Appointment.create()` completes, the patient sees confirmation, and email is sent in the background.

**Correct pattern:**
```javascript
// CORRECT — fire and forget
const appointment = await Appointment.create({...});
emailService.sendBookingConfirmation(appointment); // No await
res.json(apiResponse.success(appointment, 'Appointment booked'));
```

The email function is not awaited. It may fail silently, but the patient's booking succeeds regardless.

---

## Consequences

1. **Email delivery is eventually consistent:** Emails may arrive seconds or minutes after the appointment is created. Users must expect this. The in-app notification is immediate; the email is eventual.

2. **Email failures are not user-facing:** If an email fails to send, the patient does not see an error. However, the admin can see email delivery logs and resend manually if needed.

3. **No error propagation:** Email service errors do not bubble up to the booking controller. Errors are logged to a monitoring service (e.g., Sentry), not caught by the API.

4. **Job queue recommended:** For production reliability, a job queue (Bull, RabbitMQ) should handle email jobs. This ensures reliability without awaiting.

5. **Email idempotency:** Email service must be idempotent. Sending the same email twice (due to retry logic) should not result in duplicate emails. Use deduplication by appointment ID.

6. **Transactional emails only:** This pattern applies to transactional emails (bookings, payments, confirmations). Marketing emails may use different logic.

---

## Alternatives Considered

**Approach 1: Await email, retry on timeout**
```javascript
await emailService.sendBookingConfirmation(appointment, { retries: 3, timeout: 5000 });
```

*Rejected:* Still blocks patient. Retry logic adds latency. Email should not block business logic.

**Approach 2: Queue email, await queue confirmation**
```javascript
await emailQueue.add(appointment);
// Queue immediately confirms, then processes in background
```

*Rejected:* More infrastructure. Current approach is simpler.

**Approach 3: Send email synchronously, log failures**
```javascript
try {
  await emailService.sendBookingConfirmation(appointment);
} catch (err) {
  logger.error('Email failed', err);
}
```

*Rejected:* Still blocks patient if email service is slow. Try/catch is good for monitoring, but goal is to not await.

## Implementation Notes

- Email function does not use `await` in controller
- Email service function accepts appointment object and sends mail
- Errors are caught inside email service, logged to monitoring
- No `throw` from email service that would propagate to controller
- Email template should include appointment ID for deduplication
- Tests: Verify booking completes even if email service fails
- Monitoring: Alert if email failure rate exceeds 5%

**Example email service:**
```javascript
export async function sendBookingConfirmation(appointment) {
  try {
    await mailer.send({
      to: appointment.patientEmail,
      subject: `Appointment Confirmed`,
      template: 'booking-confirmation',
      data: appointment,
      idempotencyKey: appointment._id // Prevent duplicates
    });
  } catch (err) {
    logger.error('Email send failed', { appointmentId: appointment._id, err });
    Sentry.captureException(err);
  }
}
```

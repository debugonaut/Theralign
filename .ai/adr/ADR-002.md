# ADR-002 — Separate Payment Collection from Appointment

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

In phase 1-5, the Payment model could have been embedded as a nested document inside the Appointment collection. However, appointments and payments have different lifecycles and different access patterns:

- **Appointments:** Queried by patient, doctor, date. Updated by appointment cancellation, completion status. Displayed in appointment lists.
- **Payments:** Queried by Razorpay webhook verification. Updated by refund workflow. Accessed by admin payment dashboard.

Embedding payments in appointments would require:
- Larger Appointment documents (every appointment carries payment data)
- Separate query indexes on nested payment fields
- Complex partial updates when only payment status changes
- Inconsistent query patterns (sometimes fetch appointment with payment, sometimes payment alone)

---

## Decision

Payment is a separate top-level collection, linked to Appointment via `appointment` ObjectId field.

**Appointment model:**
```javascript
appointment: {
  type: mongoose.Schema.Types.ObjectId,
  ref: 'Appointment'
}
```

**Payment model:**
```javascript
appointment: {
  type: mongoose.Schema.Types.ObjectId,
  ref: 'Appointment',
  required: true,
}
```

Payment queries are independent. Appointments are queried without assuming payment exists.

---

## Consequences

1. **Retrieval patterns change:**
   - To fetch appointment with payment: `await Appointment.findById(id).populate('paymentId')` → `await Payment.findOne({ appointment: id })`
   - This is explicit, not implicit

2. **Financial snapshot:** Payment and Appointment each store their own copies of `platformCommission` and `doctorEarnings`. This is not data duplication — it is data immutability. Each document captures the state as it was at the time of the transaction.

3. **Consistency:** Payment creation is not required before appointment creation. However, the booking flow enforces: appointment must transition to `confirmed` only after payment transitions to `paid`.

4. **Admin analytics:** Admin queries aggregate across both collections:
   ```javascript
   // Revenue from all paid appointments
   const revenue = await Payment.aggregate([
     { $match: { status: 'paid' } },
     { $group: { _id: null, total: { $sum: '$amount' } } }
   ]);
   ```

5. **Refund workflow:** Refund state is entirely in the Payment collection. Appointment cancellation triggers refund logic, but refund status lives on Payment, not Appointment.

6. **Performance:** Smaller Appointment documents mean faster queries. Payment queries use index on `appointment` field.

---

## Alternatives Considered

**Approach 1: Embed Payment in Appointment**
Store full payment data as nested document in Appointment.

*Rejected:* Makes Appointment documents larger. Payment updates require full Appointment write. Complicates Razorpay webhook handling (must fetch Appointment first, then update nested Payment field).

**Approach 2: Embed Appointment reference in Payment, with Payment array in Appointment**
Store Payment[] as nested array in Appointment, plus appointment reference in Payment.

*Rejected:* Worse than embedding full payment. Array growth over time as payment history adds records.

**Approach 3: Single Payment document, multiple Appointment references**
One Payment for multiple Appointments (e.g., bundled booking).

*Rejected:* Theralign requires one-to-one relationship. Over-engineered.

## Implementation Notes

- Payment creation happens after Appointment creation
- Razorpay webhook handlers query Payment by razorpayOrderId
- Admin payment dashboard queries Payment collection directly, not through Appointment
- Indexes: `{ appointment: 1 }`, `{ patient: 1 }`, `{ razorpayOrderId: 1, unique: true }`

# ADR-005 — Indian Number Formatting (en-IN Locale)

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Theralign is an Indian SaaS platform serving Indian patients and physiotherapists. Financial amounts, phone numbers, and dates are displayed to Indian users.

Indian formatting conventions differ from US conventions:
- **Numbers:** 1,23,45,678 (Indian system) vs. 123,456,789 (Western system)
- **Currency:** ₹1,23,456 vs. $123,456
- **Phone:** +91 98765 43210 vs. +91-9876543210
- **Date:** DD/MM/YYYY (13/06/2026) vs. MM/DD/YYYY (06/13/2026)

Using Western formatting in an Indian app looks like a bug to users. It also violates accessibility — screen readers read numbers differently across locales.

---

## Decision

All formatting uses the `en-IN` locale (English, India) via JavaScript Internationalization API (`Intl`).

**Numbers:**
```javascript
const formatter = new Intl.NumberFormat('en-IN');
console.log(formatter.format(123456789)); // "1,23,45,789"
```

**Currency (Rupees):**
```javascript
const currencyFormatter = new Intl.NumberFormat('en-IN', {
  style: 'currency',
  currency: 'INR'
});
console.log(currencyFormatter.format(500)); // "₹500.00"
```

**Phone numbers:**
Use format `+91 XXXXX XXXXX` for display (hyphen + space pattern).

**Dates:**
Use format `DD/MM/YYYY` for display (e.g., 13/06/2026).

---

## Consequences

1. **Frontend formatting:** All monetary displays in React components use `Intl.NumberFormat('en-IN')`.

2. **Backend formatting:** Backend does not format numbers. Formatting is the frontend's responsibility. Backend sends raw numbers in JSON responses.

3. **Database storage:** Database stores numbers as integers (rupees, not formatted strings). No formatting in MongoDB.

4. **Phone display:** Phone numbers are stored in E.164 format (+919876543210) but displayed as +91 98765 43210.

5. **Date display:** Dates are stored as YYYY-MM-DD strings in database, displayed as DD/MM/YYYY in UI.

6. **Backwards compatibility:** Older Western-formatted systems cannot be bulk-imported without reformatting. This is accepted as the cost of localization.

---

## Alternatives Considered

**Approach 1: Use US locale for internal consistency**
Store and display all numbers in Western format, only translate to Indian format at export time.

*Rejected:* Users see Western formatting in app, which is confusing and unprofessional.

**Approach 2: Custom formatting functions**
Write custom JS functions to format Indian numbers instead of using `Intl` API.

*Rejected:* `Intl` is standard, performant, and maintained. Custom code introduces bugs and maintenance burden.

**Approach 3: Format in backend, send formatted strings**
Backend sends pre-formatted strings like "₹1,23,456" instead of raw numbers.

*Rejected:* Breaks API contract. API should return data, not presentation. Frontend handles presentation.

## Implementation Notes

- Frontend utility function: `formatCurrency(amount)` wraps Intl.NumberFormat
- Frontend utility function: `formatPhone(number)` applies +91 formatting
- Frontend utility function: `formatDate(dateString)` converts YYYY-MM-DD to DD/MM/YYYY
- Backend: Returns all numbers as integers, all dates as YYYY-MM-DD strings
- Tests: Verify formatting for edge cases (0, 1000000, etc.)
- Component example:
  ```jsx
  import { formatCurrency } from '@/utils/formatting';
  
  export function PriceDisplay({ amount }) {
    return <span>{formatCurrency(amount)}</span>;
    // Renders: ₹500.00
  }
  ```

# ADR-004 — Consultation Fee Snapshotted at Booking Time

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Doctors can update their consultation fee at any time. If consultation fees were recalculated dynamically, changing a fee would retroactively alter:
- Historical appointment records visible to patients
- Revenue calculations in admin analytics
- Amounts owed to doctors

Patients booked appointments when the consultation fee was $500. If the doctor later raises the fee to $750, the patient-visible appointment should still show $500, not $750. Same for admin analytics — reported revenue must match actual cash movement.

---

## Decision

The `consultationFee`, `platformCommission`, and `doctorEarnings` are written to the Appointment record at booking time by reading the current DoctorProfile and calculating the split. These values are **never recalculated** after the appointment is created.

**At booking time:**
```javascript
const doctor = await DoctorProfile.findById(doctorId);
const appointment = await Appointment.create({
  consultationFee: doctor.consultationFee,  // Snapshot
  platformCommission: Math.floor(doctor.consultationFee * 0.10),
  doctorEarnings: Math.floor(doctor.consultationFee * 0.90),
  // ...
});
```

**Later, never read from DoctorProfile:**
```javascript
// WRONG
const currentFee = await DoctorProfile.findById(doctorId).select('consultationFee');

// CORRECT
const snapshot = await Appointment.findById(appointmentId);
const fee = snapshot.consultationFee;
```

---

## Consequences

1. **Display code reads from Appointment:** Any page that displays appointment fees (patient dashboard, doctor dashboard, admin analytics) reads from Appointment, not DoctorProfile.

2. **Revenue aggregation:** Admin analytics sum `Appointment.platformCommission` and `Appointment.doctorEarnings`, not recalculating from current fees.

3. **Doctor earnings calculation:** When generating payout reports, use `Appointment.doctorEarnings`, not recalculating from current fee.

4. **Payment snapshot:** Payment model also stores copies of these fields at payment time:
   ```javascript
   const payment = await Payment.create({
     platformCommission: appointment.platformCommission,
     doctorEarnings: appointment.doctorEarnings,
     // ...
   });
   ```

5. **Historical accuracy:** Changing a doctor's fee in the future does not affect past appointments.

6. **Refund consistency:** If a refund is issued for an appointment, the refund amount matches the original appointment fee, not the current fee.

---

## Alternatives Considered

**Approach 1: Recalculate fees dynamically**
Store only appointment reference. Calculate fees at display time from current DoctorProfile fee.

*Rejected:* Breaks historical accuracy. Admin would report revenue that differs from actual cash received. Patients see their appointment fee change retroactively.

**Approach 2: Store fee in separate PriceHistory collection**
Create a PriceHistory collection with timestamped fee entries. Look up fee at appointment date.

*Rejected:* Over-engineered. Requires storing date on every fee entry, migrating old fee entries when doctors change fees, and complex queries on PriceHistory. A snapshot field on Appointment is simpler.

**Approach 3: Freeze DoctorProfile fee version**
Store the fee on DoctorProfile with a version timestamp. Look up versioned fees at appointment time.

*Rejected:* Requires querying historical versions. Snapshot field is simpler.

## Implementation Notes

- Booking controller: Copy fees from DoctorProfile to Appointment
- Admin analytics: Query Appointment fields, not DoctorProfile
- Doctor dashboard: Display Appointment.doctorEarnings for historical earnings
- Refund service: Use Appointment fee snapshot for refund calculation
- Tests: Verify fee change on DoctorProfile does not affect existing Appointments
- Tests: Verify Payment snapshot matches Appointment snapshot at creation time

# ADR-007 — Session Records as Separate Collection from Appointment

**Status:** Accepted

**Date:** 2026-06-11

---

## Context

Phase 14 introduces session records — the clinical notes and prescribed exercises that a doctor documents during or after an appointment. This data has different access patterns and retention requirements than appointment records:

- **Appointment:** Queried by patient and doctor, displayed in booking lists, used for refund workflow
- **Session Record:** Queried by doctor only (privacy), long-term storage for medical history, modified independently of appointment status

If session records were embedded in Appointment:
- Appointment documents would grow over time (if exercises are added/updated)
- Patient-facing appointment queries would carry session data that patients may not want retrieved
- Deleting a session record would require partial document updates on Appointment

---

## Decision

SessionRecord is a separate top-level collection, linked to Appointment via `appointment` ObjectId field.

**Appointment model:** No embedded session data.

**SessionRecord model:**
```javascript
{
  appointment: ObjectId(Appointment),
  doctor: ObjectId(DoctorProfile),
  patient: ObjectId(User),
  
  clinicalNotes: String,
  prescribedExercises: [
    { exercise: String, duration: String, frequency: String }
  ],
  
  documentUrl: String, // PDF uploaded to Cloudinary
  
  createdAt: Date,
  updatedAt: Date
}
```

---

## Consequences

1. **Separate queries:** Retrieving appointment does not automatically fetch session record. Explicit populate required.

2. **Doctor-only access:** Session records are private to the doctor. Patients can see they exist but not read detailed content. Authorization enforced at route level.

3. **Long-term storage:** Session records may outlive appointments. A completed, refunded appointment may have its SessionRecord archived separately.

4. **Modification independence:** Doctor can add/update session notes after appointment is completed. Appointment record remains unchanged.

5. **Optional session record:** Not all appointments require a session record. Doctor may choose not to document (not recommended, but system allows it).

6. **Concurrent updates:** Multiple doctors updating different fields on same appointment do not interfere with SessionRecord creation.

---

## Alternatives Considered

**Approach 1: Embed SessionRecord in Appointment**
Store clinical notes and exercises as nested object in Appointment.

*Rejected:* Violates separation of concerns. Appointment and SessionRecord have different owners (patient can see appointment, only doctor can see session). Embedding forces sharing authorization.

**Approach 2: Single generic Document collection for all documents**
Create a Document collection with type: 'appointment', type: 'session', type: 'credential', etc.

*Rejected:* Too generic. Different document types have very different schemas and access patterns.

**Approach 3: Store session data in a separate service (external CMS, EHR)**
Use specialized medical records system.

*Rejected:* Over-engineering for MVP. SessionRecord is simple structured data.

## Implementation Notes

- SessionRecord model: `server/src/models/SessionRecord.model.js`
- SessionRecord routes: `server/src/routes/sessionRecord.routes.js`
- SessionRecord controller: `server/src/controllers/sessionRecord.controller.js`
- Authorization: Only treating doctor can create/update SessionRecord
- Indexes: `{ appointment: 1 }`, `{ doctor: 1 }`, `{ patient: 1 }`
- Tests: Verify patient cannot read SessionRecord, only doctor can
- API endpoints (Phase 14):
  - POST /api/session-records/:appointmentId — Create session record
  - GET /api/session-records/mine — Doctor's session records
  - PUT /api/session-records/:id — Update session record
